# kb-update — 지식베이스 갱신 (라우팅 + 정본 + 시각자료)

## 0. 먼저 규칙을 읽는다 (필수)

`${CLAUDE_PLUGIN_ROOT}/skills/kb-conventions/references/conventions.md`를 Read한다.

## 1. 대상 파악

- **프로젝트**: 인자/대화에서 식별. 모르면 `<vault>` 하위 폴더 목록을 보여주고 고르게 한다.
- 프로젝트 폴더가 없으면 `kb-init`을 먼저 안내(여기서 새 구조를 만들지 않는다).

## 2. 라우팅 (어디에 넣을지)

`conventions.md`의 영역 매핑으로 내용을 분류한다:
- 할 일/추후 과제/미착수 아이디어 → `00. 백로그`
- 개요/설계 → `1. 개요·전략`
- 계획/아이디어/현황/흐름 → `2. 개발·구현`
- 데이터/벤치마크/비교·조사 분석 → `3. 분석자료`
- 실행/운영 → `4. 운영·셋업`
- 사용자 매뉴얼 → `5. 사용설명서`
- 논의 중인 내용/미결 결정/오픈 퀘스천 → `90. 논의사항`
- 코드리뷰 결과/리뷰 코멘트 정리 → `91. 코드리뷰`
- 다른 작업 중 발견했으나 지금 처리하지 않는 결함/이슈 → `92. 발견된 이슈` (이슈 1건=문서 1개, `templates/doc-stubs/발견된 이슈.md` 스텁)

적합한 기존 문서가 있으면 그 문서의 **해당 섹션**에 넣는다. 없으면 새 문서를 어디에 만들지 사용자에게 제안·확인한다. 작업성 폴더(`00`·`90`·`91`·`92`)에 첫 문서를 만들 때는 빈 폴더(`.gitkeep`만 있는 상태)일 수 있으므로 폴더 존재를 먼저 확인한다.

## 2-1. 칸반 카드 생성 (설계문서)

기능/작업 단위 **설계문서 카드**를 만들라는 요청이면(예: "이 기능 설계문서 만들어", "구현 카드 추가") `conventions.md` §11을 따른다:

- 대상 위치는 항상 `2. 개발·구현/칸반보드/1. 구현 전/`. 컬럼 폴더가 없으면 `kb-init`을 먼저 안내한다(여기서 보드 구조를 새로 만들지 않는다).
- `templates/doc-stubs/설계문서.md`를 복사해 카드를 만든다. 파일명은 한글 제목 그대로(§2). frontmatter `상태: 구현 전`, `{{CARD_TITLE}}`·`{{ONE_LINE_DEFINITION}}` 등 placeholder를 채운다.
- **`칸반보드.md` 갱신**: `## 1. 구현 전` 컬럼에 `- [ ] [[카드 제목]]`을 추가(부분 갱신).
- 관련 standing 문서([[구현 계획]]·[[아이디어 백로그]])에서 이어지면 `[[wikilink]]`로 연결한다. 카드를 만들었으면 §6에서 MOC에도 링크를 건다.

> 카드의 **상태 이동**(구현 시작/완료/리오픈)은 이 스킬이 아니라 `kb-status`가 한다.

## 3. 정본(canonical) 검사

- 이 주제의 정본이 MOC 정본 테이블에 이미 있는지 확인.
- 있으면 **중복 정본을 새로 만들지 않는다.** 기존 정본을 갱신하거나, 새 문서가 정본을 `[[링크]]`로 가리키게 한다.
- 새로운 정본이 필요하면 MOC 정본 테이블에 한 줄 추가.

## 4. 쓰기 (부분 갱신)

- 대상 문서를 **Read → 해당 섹션만 Edit/갱신**한다. 통째 덮어쓰기 금지.
- frontmatter의 `date`를 오늘로 갱신.
- 새 문서를 만들면 적절한 스텁/형식과 frontmatter(tags 포함)를 갖춘다.
- 관련 문서에 `[[wikilink]]`를 건다.

## 5. 시각 자료 (흐름성 문서는 기본 포함)

- **흐름·구조·아키텍처·라이프사이클(호출 시점)·시퀀스·경로 분기를 설명하는 문서는 다이어그램을 기본으로 포함**한다(선택 아님 — `conventions.md` §7). 글로만 흐름을 풀지 말고, 도입부 직후에 전체 흐름 SVG를 한 장 임베드(`![[파일.svg]]`)하고 글은 그 그림의 "왜·어디(코드 위치)"를 보충한다. 단순 백로그·메모·논의 노트엔 강제하지 않는다.
- 흐름/구조/시퀀스/라이프사이클을 추가·변경하면 **SVG를 `<프로젝트>/images/`에 생성**한다. 큰 흐름의 한 단계가 복잡하면 그 박스를 확대한 상세 SVG를 해당 섹션에 덧붙인다(개요도 + 상세도 짝).
- **여러 패키지·모듈·계층이 얽힌 흐름은 패키지별로 색을 구분**해 그리고, 색=패키지 매핑을 범례로 명시한다(`diagram-design.md` §7.1).
- **Mermaid 금지**: 도식은 크기·복잡도와 무관하게 SVG 로만 만든다. 본문에 ```mermaid 블록을 두지 않는다(`conventions.md` §7). 기존 문서에 ```mermaid 가 있으면 SVG 로 교체한다.
- SVG 작성은 `conventions.md` §7(저장·임베드)과 `references/diagram-design.md`(색 토큰·**패키지별 색 구분**·노드 모양·종단점 강조·**흐름도 패턴/라이프사이클 고유 장치**·픽토그램·가독성·테마 안전)를 따른다. 그리기 전후로 `diagram-design.md` §9 체크리스트를 점검한다.

## 6. MOC 갱신 + 정합성

- 새 문서를 만들었으면 MOC 영역 목록에 `[[링크]]`를 추가.
- 진행 현황이 바뀌면 MOC `[!success]` 블록 갱신.
- 끝에 **`kb-curate`를 호출**해 링크·이미지·정본·카탈로그 정합성을 점검한다.
