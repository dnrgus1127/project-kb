# 프로젝트 지식베이스 공통 규칙 (conventions)

> 이 파일이 규칙의 **유일한 출처**다. 모든 kb-* 스킬·에이전트가 작업 전에 이 파일을 Read한다.
> 규칙을 바꾸려면 이 파일만 고친다. 다른 곳에 복사하지 않는다.

---

## 0. 보관함 루트(vault root) 해석

지식 문서는 Obsidian 보관함 아래에 저장한다. 경로는 **하드코딩하지 않는다**.

### config 파일 (영속 저장 위치)

보관함 루트 절대경로는 **플러그인 설치 환경별로 한 번 입력받아 config 파일에 영속 저장**한다.

```
${HOME}/.claude/project-kb/config.json
```

- Windows에서는 `%USERPROFILE%\.claude\project-kb\config.json`.
- 스키마:
  ```json
  { "vaultRoot": "<보관함 루트 절대경로>" }
  ```
  - 예(Windows): `{ "vaultRoot": "C:\\WebOfficeEnvironment\\docs" }`
- **버전이 박힌 캐시 경로(`.../project-kb/<version>/`) 안에는 저장하지 않는다.** 플러그인 업데이트 시 버전 세그먼트가 바뀌면 캐시 내부 설정이 날아가기 때문이다. 위 `~/.claude/project-kb/` 경로는 버전과 무관하게 유지된다.

### 해석 순서

1. **config 파일**(`~/.claude/project-kb/config.json`)의 `vaultRoot`가 있으면 그 값을 쓴다 — 우선.
2. 없으면 환경변수 `KB_VAULT_ROOT` (하위 호환).
3. 둘 다 없으면 사용자에게 보관함 루트 **절대경로를 묻고**, 존재를 확인한 뒤 **config 파일(`~/.claude/project-kb/config.json`)에 영속 저장**하고 사용한다. (config 디렉터리/파일이 없으면 만든다. 이 config 파일 자체는 보관함이 아니므로 생성해도 된다.)

규칙 표기에서 `<vault>` = 위 해석으로 정해진 보관함 루트 절대경로. 즉 프로젝트는 `<vault>/<프로젝트>/`에 생성된다.

> 보관함 루트 절대경로를 **나중에 변경**하려면 `kb-set-vault` 스킬을 쓴다(현재 config를 보여주고, 새 절대경로를 검증 후 config 파일에 기록).

**가드**: 보관함 폴더를 자동 생성하지 않는다. 대상 폴더가 이미 있으면 사용자에게 확인하고 진행한다. 새 폴더(특히 보관함 루트)가 없으면 만들기 전에 확인한다. (단, config 파일 `~/.claude/project-kb/config.json` 자체는 보관함이 아니므로 이 가드 대상이 아니다 — 필요하면 디렉터리·파일을 생성한다.)

## 1. 프로젝트 폴더 taxonomy

```
<vault>/
├── 📚 프로젝트 카탈로그.md         # 전 프로젝트 횡단 인덱스 (루트 카탈로그)
├── _archive/                       # 종료/중단 프로젝트 보관
└── <프로젝트>/
    ├── 📑 인덱스.md                # MOC (이 프로젝트의 지도)
    ├── 1. 개요·전략/               # 무엇을·왜  — 코드 무관한 순수 설계/개요
    ├── 2. 개발·구현/               # 어떻게 만드는가 — 계획·아이디어·구현현황·흐름도
    ├── 3. 운영·셋업/               # 어떻게 띄우고 돌리는가 — 실행/프로세스/원격
    ├── 4. 사용설명서/              # 나에게 무엇을 해주는가 — 최종 사용자 매뉴얼
    └── images/                     # 시각 자료(SVG 등)를 한곳에 모음
```

### 영역별 표준 문서 (사용자 요청 매핑)
| 영역 | 표준 문서(예시) | 사용자 요청 매핑 |
|---|---|---|
| 1. 개요·전략 | `개요.md`, (도메인 설계 문서들) | **개요** |
| 2. 개발·구현 | `구현 계획.md`, `아이디어 백로그.md`, `구현 현황.md`(정본), `데이터 흐름도.md` | **구현 계획 / 추가 구현 계획·아이디어 / 현재 구현 상태 / 데이터 흐름도** |
| 3. 운영·셋업 | `운영 가이드.md`, 실행 명령 레퍼런스 | **운영 방법** |
| 4. 사용설명서 | `00 사용설명서 개요.md`, 화면/알림/데이터 설명 | **사용 설명** |

프로젝트 성격에 따라 영역 1의 세부 문서는 달라질 수 있다(예: 투자전략이면 전략 설계 문서들). 영역 폴더 4개와 `images/`, MOC는 **항상** 둔다.

## 2. 파일·폴더 명명

- 영역 폴더: 번호 접두 + 한글 이름 (`1. 개요·전략`). 번호로 정렬 보장.
- 문서 파일: 사람이 읽는 한글 제목 그대로 (`구현 계획.md`). 확장자 `.md`.
- 특수 문서: 이모지 접두 허용 (`📑 인덱스.md`, `📚 프로젝트 카탈로그.md`).
- 이미지: `images/` 안에 내용을 드러내는 한글/kebab 이름 (`데이터 흐름도.svg`, `시장감시 시퀀스.svg`).
- 한글·공백·이모지 경로를 다룰 때 셸을 쓰면 **반드시 따옴표**로 감싼다. 가급적 Claude 파일 도구로 직접 다룬다.

## 3. Frontmatter

모든 문서 첫머리에 YAML frontmatter:

```yaml
---
tags: [차원, 용도, 프로젝트슬러그]
date: YYYY-MM-DD
---
```

- **tags**:
  - 차원: 도메인 성격(예: `투자전략`, `백엔드`, `데이터`)
  - 용도: `MOC` · `개요` · `계획` · `아키텍처` · `현황` · `가이드` · `사용설명서` · `아이디어`
  - 프로젝트슬러그: 공백 없는 프로젝트 식별자(예: `주도주낙폭과대전략`)
- **date**: 마지막 의미 있는 갱신 날짜. 갱신 시 함께 올린다.

## 4. Wikilink

- 문서 간 링크는 **파일명 기반** `[[제목]]` (확장자·경로 없음). Obsidian이 파일명으로 해석.
- 링크를 적극적으로 건다 — 관련 문서, 정본, 코드 리포 문서를 서로 연결.
- 리네임/이동 시 링크가 깨지므로, 변경 후 `kb-curate`로 broken-link/orphan을 점검한다.

---

## 5. Callout (Obsidian)

핵심 정보를 callout으로 강조:

```
> [!info] 한 줄 정의
> [!success] 진행 현황 (YYYY-MM-DD)
> [!warning] 주의/리스크
> [!note] 보조 설명
> [!tip] 팁
```

MOC에는 최소 `[!info] 한 줄 정의`와 `[!success] 진행 현황`을 둔다.

## 6. 정본(canonical) 규칙

같은 주제가 여러 문서에 걸치면, **기준 문서(정본) 1개**를 정하고 나머지는 정본을 가리킨다.

- 정본 매핑은 각 프로젝트 MOC의 **정본 테이블**에 둔다(주제 → 정본 → 함께 보는 문서).
- 새 문서를 추가할 때 `kb-update`는 "이 주제의 정본이 이미 있는가?"를 먼저 확인한다. 있으면 **중복 정본을 만들지 않고** 기존 정본을 갱신하거나 새 문서가 정본을 가리키게 한다.
- **코드 vs 문서**: 문서가 코드 동작과 충돌하면 **코드 리포가 항상 기준**이다. MOC에 코드 리포 경로를 명시한다.

## 7. 다이어그램 / 시각 자료

문서는 사람이 읽으므로, 흐름이나 구조는 글보다 그림이 낫다.

- **데이터·비즈니스 흐름도, 시스템 구성도, 의사결정 블록도, 시퀀스도** 등은 **SVG로 만들어 프로젝트 `images/` 폴더에 저장**한다.
- 문서에서 임베드:
  - Obsidian 임베드: `![[데이터 흐름도.svg]]`
  - 또는 상대경로: `![데이터 흐름도](images/데이터%20흐름도.svg)`
- 텍스트로 충분한 도식은 **Mermaid 코드펜스**(```mermaid)로 본문에 직접 둔다. 인쇄/공유가 필요하면 SVG로 내보내 `images/`에 병기.
- **드리프트 대상**: 흐름도가 가리키는 코드/규칙이 바뀌면 SVG도 갱신한다. `kb-status`가 흐름도 갱신을 제안하고, 승인 시 `images/`의 SVG를 재생성한다.
- `kb-curate`는 깨진 이미지 임베드와 어디서도 참조되지 않는 고아 SVG를 보고한다.

### SVG 작성 지침 (저장·임베드)
- 보관함은 로컬 파일이므로 외부 폰트/스크립트 의존을 피한다. `<text>`에 시스템 폰트(예: `font-family="-apple-system, sans-serif"`), 한글 그대로 사용 가능.
- `viewBox`를 지정해 확대/축소에 견디게 한다.
- 색은 의미를 담아 일관되게(예: 입력=파랑, 처리=회색, 위험/경보=빨강).

> [!tip] 시각 품질 상세는 별도 가이드
> 색 토큰·노드 종류별 모양·종단점(시작/끝) 강조·픽토그램·가독성·다크 테마 안전 등 **어떻게 그려야 읽히는지**는
> `${CLAUDE_PLUGIN_ROOT}/skills/kb-conventions/references/diagram-design.md`를 따른다.
> 흐름도/구성도/시퀀스도 SVG를 만들거나 갱신할 때 함께 Read한다.

## 8. MOC(📑 인덱스) 필수 구성

각 프로젝트 MOC에 포함:
1. 한 줄 정의 (`[!info]`)
2. 진행 현황 (`[!success]`, 날짜 포함, Phase 상태)
3. "어디부터 읽나" — 목적별 진입점 표
4. 영역별 문서 목록 (1~4 + 시각자료)
5. 정본 테이블
6. 코드 리포 경로 + "코드가 기준" 주석

템플릿: `${CLAUDE_PLUGIN_ROOT}/skills/kb-conventions/references/templates/moc.md`

## 9. 루트 카탈로그(📚 프로젝트 카탈로그)

`<vault>/📚 프로젝트 카탈로그.md`는 전 프로젝트 횡단 인덱스:
- 프로젝트별 한 줄: 이름(MOC 링크) · 상태(활성/관찰/완료/보류/아카이브) · 한 줄 요약 · 정본 진입점.
- `kb-init`이 새 프로젝트를 등록하고, `kb-archive`가 상태를 갱신하며, `kb-curate`가 정합성을 유지한다.

템플릿: `${CLAUDE_PLUGIN_ROOT}/skills/kb-conventions/references/templates/catalog.md`

## 10. 쓰기·커밋 원칙

- **쓰기 단일 경로**: 보관함 기록은 스킬만 한다. 에이전트는 초안/감사 결과만 산출한다.
- **부분 갱신**: 문서를 통째로 덮어쓰지 않는다. 현재 파일을 Read → 해당 섹션만 갱신.
- **자동 덮어쓰기 금지**: 구현현황·정본·Phase는 드리프트 리포트 → 사람 승인 후에만 반영.
- **단일 커밋**: 한 작업 단위가 끝나면 `kb-curate` 후 보관함을 한 번 커밋하도록 권고(보관함은 로컬 git).
