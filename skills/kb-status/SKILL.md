# kb-status — 코드↔문서 드리프트 감사·갱신 (승인 게이트)

## 0. 먼저 규칙을 읽는다 (필수)

`${CLAUDE_PLUGIN_ROOT}/skills/kb-conventions/references/conventions.md`를 Read한다.

## 1. 대상 파악

- **프로젝트**와 **코드 리포 경로**를 확인(인자/대화/ MOC의 코드 리포 경로). 모르면 묻는다.

## 2. 감사 (kb-status-pm 에이전트에 위임)

- `kb-status-pm` 서브에이전트를 띄워 **드리프트 리포트**를 받는다. 프롬프트에 프로젝트 폴더와 코드 리포 경로, "근거(커밋/파일) 기반, 보관함 쓰기 금지"를 명시한다.
- 리포트에는 일치/드리프트 항목, 흐름도 갱신 필요 여부, Phase 제안이 포함된다.

## 3. 승인 게이트 (중요)

- 드리프트 리포트를 사용자에게 **그대로 보여주고**, 어느 항목을 반영할지 고르게 한다(AskUserQuestion 또는 목록 확인).
- **승인되지 않은 항목은 반영하지 않는다.** 추측 기반 자동 덮어쓰기 금지.

## 4. 반영 (스킬이 쓴다)

- 승인된 항목만 [[구현 현황]]·[[구현 계획]]의 Phase·관련 문서에 **부분 갱신**(Read → 섹션 Edit). frontmatter `date` 갱신.
- 흐름도 갱신이 승인되면 `<프로젝트>/images/`의 SVG를 다시 그리고 본문 임베드를 확인. SVG는 `conventions.md` §7과 `references/diagram-design.md`(시각 품질·§8 체크리스트)를 따른다.
- MOC의 `[!success]` 진행 현황 블록을 갱신.

## 5. 정합성 + 마무리

- `kb-curate`를 호출해 링크·이미지·정본·카탈로그를 점검한다.
- 무엇을 근거로 무엇을 갱신했는지 요약 보고. 보관함 단일 커밋 권고.

> 에이전트는 읽기/감사만, 쓰기는 이 스킬만. 코드와 충돌 시 코드가 기준이지만, **문서 반영은 항상 사람 승인을 거친다.**
